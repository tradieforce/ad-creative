# The Method

> **The simple per-ad recipe we proved out through testing.** If you read nothing else, read this. Everything else in the bundle is just this method scaled up, formalised, and engineered for production.

---

## The 6-step recipe (per ad)

```
   1. ONBOARDING FORM
   ─────────────────────
   Client fills: business name, city, region, Google reviews,
   brands installed, default promo %, photos (team/owner/van).

              ↓

   2. PICK (deterministic)
   ─────────────────────
   System picks for this ad:
   - Archetype (A1–A10) — based on what fires for this client
   - Variant — based on what applies (summer/winter, etc)
   - Headline + CTA + badge — from the archetype's variable_inputs pool
   - Pricing mode (per_week / fixed_price / no_price)

              ↓

   3. ATTACH (3 things only)
   ─────────────────────
   - ONE reference image — the archetype's matched exemplar
     (e.g. A1 → A03.png, A4 → A04.png, A10 → A37.png)
   - The locked components — house diagram, condensers,
     brand logos, trust badges, etc
   - The client photo (only for A10 trust ads)

              ↓

   4. COMPOSE (Claude with vision)
   ─────────────────────
   Claude looks at every attached image. Pulls the palette
   from the client's actual photos when present (this is
   what made the Sharp test work — orange wall, blue stripe,
   navy polos drove the entire ad palette).
   Writes the ChatGPT prompt for this specific ad.

              ↓

   5. RENDER (ChatGPT gpt-image-1)
   ─────────────────────
   Receives Claude's composed prompt + the attachments.
   Generates the PNG (1080×1080 or 1080×1350 for A9 heavy).

              ↓

   6. REVIEW
   ─────────────────────
   Bray approves / regenerates / recomposes / rejects.
   Approved ads exported as zip → uploaded to Meta Ads Manager.
```

---

## That's it. That's the method.

Every successful test in the design phase used this exact flow. We tested it live across 8+ archetypes, captured what worked as gold standards (in `05_GOLD_STANDARDS/`), and refined the prompts until the outputs landed.

## Why this is "simple enough but includes everything"

**Simple enough:**
- Per ad, it's 6 steps. Step 4 is the only AI-heavy step. Step 3 is "attach 3 things."
- One reference image per ad. Not many. Not a lookup table. Just one matched exemplar.
- Locked components are uploaded once, used as-is, never regenerated by AI.
- The prompt that gets sent to ChatGPT is what we tweak when an ad doesn't land — same thing we did during testing.

**Includes everything:**
- All 10 archetypes (A1 through A10) — every angle we proved works
- Holiday + seasonal variants — generated upfront, launched as the dates apply
- Pricing variants — per-week default, one fixed-price per pack, trust ads with no price
- Vision-aware composition — pulls brand colours from the client's photos
- Pack balance — funnel coverage, single fixed-price slot, no skew
- Review queue — nothing auto-publishes; you approve before Meta upload
- The 17 hard rules — applied automatically to every ad

## Where the architecture's "complexity" actually comes from

What looks complex in the bundle (3 layers, queues, workers, database schemas) is just the **same 6-step recipe scaled to packs of 25 ads per client** with the engineering needed to:

- Run 25 ads in parallel instead of one at a time
- Handle retries when the image gen produces a typo
- Balance funnel coverage across the pack (so not every ad is the same temperature)
- Award the single fixed-price slot to whichever archetype handles it best
- Track costs, log errors, persist state across the pack lifecycle
- Surface everything in a review queue UI

None of that adds to the **per-ad method**. The recipe stays the same.

## What we're tweaking when an ad doesn't land

Exactly what we did during testing — we adjust the **Layer 2 system prompt** (the instructions that tell Claude how to compose the ChatGPT prompt). That file is in `04_LAYER2/_LAYER_2_SYSTEM_PROMPT.md`. When a generated ad isn't quite right, you capture what was wrong, add it as a worked example to that prompt, redeploy. The system gets sharper over time without code changes.

## The mental model

```
Spec (source of truth)        Onboarding form (this client)
        \                              /
         \                            /
          → Pick archetype/variant ←
                    ↓
        ONE reference + locked components + client photo
                    ↓
            Claude writes the prompt (vision)
                    ↓
              ChatGPT renders
                    ↓
              Bray approves
                    ↓
                 Meta
```

If anything in the build feels more complex than this, look hard — it should still reduce to the recipe above. Every layer, every queue, every database table is just engineering scaffolding around this method.
